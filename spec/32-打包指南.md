# 打包指南

> 适用范围：AnyBridge 的本地打包、GitHub Actions 云端多平台打包、Release 发布、自动更新与常见故障排查。

## 1. 当前打包链路

AnyBridge 使用 Tauri 2 打桌面安装包，Node.js sidecar 通过 `pkg` 单独打成平台二进制，再由 Tauri 作为 `externalBin` 放进最终安装包。

当前云端 Release workflow 位于：

- `.github/workflows/release.yml`

触发方式：

- 推送 `v*` tag，例如 `v0.2.7`
- 手动运行 `workflow_dispatch`

普通 `main` push 只会触发 CI，不会自动生成 Release 安装包。

当前自动更新地址配置在 `src-tauri/tauri.conf.json`：

```text
https://github.com/soulvon/AnyBridge/releases/latest/download/latest.json
```

也就是说自动更新以 `soulvon/AnyBridge` 主仓库的 latest release 为准。发版后必须确认主仓库 Release 已发布，不能停留在 draft。

## 2. 发版前版本同步

每次发版都要同步这些文件的版本号：

- `package.json`
- `package-lock.json`
- `sidecar/package.json`
- `sidecar/package-lock.json`
- `src-tauri/Cargo.toml`
- `src-tauri/Cargo.lock`
- `src-tauri/tauri.conf.json`
- `CHANGELOG.md`

检查命令示例：

```powershell
rg '0\.2\.6|0\.2\.7' package.json package-lock.json sidecar/package.json sidecar/package-lock.json src-tauri/Cargo.toml src-tauri/Cargo.lock src-tauri/tauri.conf.json CHANGELOG.md
```

注意：

- `Cargo.lock` 里可能存在依赖自己的同版本号，例如 `displaydoc 0.2.6`，不要误改依赖版本。
- `CHANGELOG.md` 的对应版本段会被抽取成 Release Notes，并写入 `latest.json.notes`。
- 公开更新说明必须使用中文用户向文案，不要写 CI、打包链路、内部清理、密钥、仓库维护等内部信息。

## 3. 本地打包前检查

先确认工作区状态：

```powershell
git status --short --branch
```

安装依赖：

```powershell
npm ci
npm ci --prefix sidecar
```

运行基础检查：

```powershell
npm run check:ui
cargo fmt --manifest-path src-tauri/Cargo.toml --check
cargo check --manifest-path src-tauri/Cargo.toml
git diff --check
```

如果 `cargo check` 报 Tauri `externalBin` 不存在，需要先构建当前平台 sidecar。CI 里也必须先构建 sidecar 再 `cargo check`。

## 4. Sidecar 打包规则

Tauri 配置：

```json
"externalBin": ["binaries/anybridge-proxy"]
```

Tauri 实际查找文件时会自动加目标平台 triple，因此 sidecar 文件名必须是：

```text
src-tauri/binaries/anybridge-proxy-<target-triple>
```

Windows 还需要 `.exe` 后缀。

常用平台：

```text
x86_64-pc-windows-msvc
x86_64-apple-darwin
aarch64-apple-darwin
x86_64-unknown-linux-gnu
```

本地构建 Windows sidecar：

```powershell
python scripts/build/build_sidecar_plain.py --platform x86_64-pc-windows-msvc
```

云端 Release workflow 会按矩阵分别构建：

- Windows x64: `anybridge-proxy-x86_64-pc-windows-msvc.exe`
- macOS Intel: `anybridge-proxy-x86_64-apple-darwin`
- macOS Apple Silicon: `anybridge-proxy-aarch64-apple-darwin`
- Linux x64: `anybridge-proxy-x86_64-unknown-linux-gnu`

经验结论：

- Release workflow 里一定要先构建 sidecar，再调用 `tauri-action`。
- CI 里也要先构建当前 runner host triple 的 sidecar，再执行 `cargo check`。
- 缺 sidecar 时，Linux 常见错误是 `resource path 'binaries/anybridge-proxy-x86_64-unknown-linux-gnu' doesn't exist`。

## 5. 本地 Windows 安装包验证

先构建 sidecar：

```powershell
python scripts/build/build_sidecar_plain.py --platform x86_64-pc-windows-msvc
```

只打 Windows NSIS 包：

```powershell
npx tauri build --config src-tauri/tauri.nsis-only.conf.json
```

预期产物：

```text
src-tauri/target/release/bundle/nsis/AnyBridge_<version>_x64-setup.exe
```

检查生成的 NSIS 脚本是否包含安装前 hook：

```powershell
Select-String -Path src-tauri\target\release\nsis\x64\installer.nsi -Pattern 'installer-hooks|NSIS_HOOK_PREINSTALL|anybridge-proxy.exe|Copy external binaries' -Context 1,3
```

必须确认：

- `installer-hooks.nsh` 被 include
- `NSIS_HOOK_PREINSTALL` 在复制 `anybridge-proxy.exe` 前执行
- `File /oname=anybridge-proxy.exe` 使用的是当前版本 sidecar

## 6. Windows 自动更新覆盖安装经验

问题现象：

```text
无法写入 C:\Users\<user>\AppData\Local\AnyBridge\anybridge-proxy.exe
```

根因：

- 自动更新是在 AnyBridge 运行中触发的。
- 本地代理 sidecar `anybridge-proxy.exe` 仍在运行。
- Windows 锁住正在运行的 exe，NSIS 无法覆盖写入。

当前修复点：

- `src-tauri/src/commands/update.rs` 在安装更新前调用 `stop_sidecar_for_update`。
- `src-tauri/src/commands/proxy.rs` 会停止已托管代理，再按进程名清理残留 sidecar，并校验进程已退出。
- `src-tauri/windows/nsis/installer-hooks.nsh` 在 Windows 安装包复制文件前清理 `anybridge-proxy.exe` 和旧名 `ide-byok-proxy.exe`。
- `src-tauri/tauri.conf.json` 的 `bundle.windows.nsis.installerHooks` 必须指向该 hook 文件。

为什么需要两层修复：

- 应用内修复只能影响“已经安装了新版客户端之后”的后续更新。
- 如果用户当前运行的是旧客户端，真正执行覆盖安装的是新下载的安装包，因此新安装包自己也必须在安装前清理 sidecar。

## 7. 云端 Release 发版流程

推荐流程：

```powershell
git status --short --branch
npm run check:ui
cargo fmt --manifest-path src-tauri/Cargo.toml --check
cargo check --manifest-path src-tauri/Cargo.toml
git diff --check
git add <changed-files>
git commit -m "fix(updater): stop sidecar before installing updates"
git tag v0.2.7
git push origin main
git push origin v0.2.7
```

推送 tag 后检查 Actions：

```powershell
gh run list --limit 12 --json databaseId,workflowName,headBranch,status,conclusion,createdAt,displayTitle
```

查看指定 Release run：

```powershell
gh run view <run-id> --json status,conclusion,jobs --jq '{status,conclusion,jobs:[.jobs[]|{name,status,conclusion}]}'
```

Release workflow 正常阶段：

1. `Prepare draft release`
2. Windows build
3. Linux build
4. macOS x64 build
5. macOS arm64 build
6. `Rebuild latest.json and publish release`

最终必须是全部 success。

## 8. 云端产物验收

检查主仓库 Release：

```powershell
gh release view v0.2.7 --repo soulvon/AnyBridge --json tagName,isDraft,isPrerelease,publishedAt,assets --jq '{tagName,isDraft,isPrerelease,publishedAt,assetCount:(.assets|length),assets:[.assets[].name]}'
```

必须确认：

- `isDraft` 是 `false`
- `isPrerelease` 是 `false`
- `assetCount` 符合预期
- 包含 `latest.json`

当前全平台预期资产通常为 17 个：

- `AnyBridge_<version>_x64-setup.exe`
- `AnyBridge_<version>_x64-setup.exe.sig`
- `AnyBridge_<version>_x64_en-US.msi`
- `AnyBridge_<version>_x64_en-US.msi.sig`
- `AnyBridge_<version>_x64.dmg`
- `AnyBridge_<version>_aarch64.dmg`
- `AnyBridge_x64.app.tar.gz`
- `AnyBridge_x64.app.tar.gz.sig`
- `AnyBridge_aarch64.app.tar.gz`
- `AnyBridge_aarch64.app.tar.gz.sig`
- `AnyBridge_<version>_amd64.AppImage`
- `AnyBridge_<version>_amd64.AppImage.sig`
- `AnyBridge_<version>_amd64.deb`
- `AnyBridge_<version>_amd64.deb.sig`
- `AnyBridge-<version>-1.x86_64.rpm`
- `AnyBridge-<version>-1.x86_64.rpm.sig`
- `latest.json`

如果 Release workflow 成功但网页看不到包，优先检查 `isDraft`。draft release 在普通用户视角不可见。

如果需要手动发布 draft：

```powershell
gh release edit v0.2.7 --repo soulvon/AnyBridge --draft=false --prerelease=false --latest
```

## 9. 自动更新验收

下载并检查 `latest.json`：

```powershell
$tmpDir = Join-Path $env:TEMP 'anybridge-latest-check'
New-Item -ItemType Directory -Force -Path $tmpDir | Out-Null
gh release download v0.2.7 --repo soulvon/AnyBridge --pattern latest.json --dir $tmpDir --clobber
$json = Get-Content -Raw -LiteralPath (Join-Path $tmpDir 'latest.json') | ConvertFrom-Json
$json | Select-Object version, notes, pub_date | ConvertTo-Json -Depth 5
$json.platforms.'windows-x86_64'.url
```

必须确认：

- `version` 是当前版本
- `notes` 是中文用户向说明
- `windows-x86_64.url` 指向当前版本的 `x64-setup.exe`
- 对应资产存在 `.sig`

公开更新说明不要包含：

- Secret 名称以外的任何密钥内容
- 内部 CI 细节
- “误传文件”“清理敏感资料”等内部维护描述
- 私有仓库、临时脚本、调试路径
- 与用户无关的 Cargo/npm/Tauri 元数据同步细节

可以写：

- 修复了什么用户可感知问题
- 哪些平台安装体验改进
- 是否需要重启
- 是否影响自动更新

## 10. 常见故障排查

### 10.1 main 已推送但没云端包

原因：

- Release workflow 只在 `v*` tag 或手动触发时运行。
- 普通 `main` push 只跑 CI。

处理：

```powershell
git tag v<version>
git push origin v<version>
```

### 10.2 Actions 成功但 GitHub 页面看不到 Release

原因：

- Release 仍是 draft。

检查：

```powershell
gh release view v<version> --repo soulvon/AnyBridge --json isDraft,assets
```

处理：

```powershell
gh release edit v<version> --repo soulvon/AnyBridge --draft=false --prerelease=false --latest
```

### 10.3 latest.json 没更新

检查：

- `Rebuild latest.json and publish release` job 是否成功
- Release 里是否有 `latest.json`
- `latest.json.version` 是否等于当前版本
- updater endpoint 是否指向当前仓库

### 10.4 Windows 自动更新覆盖安装失败

现象：

```text
无法写入 anybridge-proxy.exe
```

处理：

- 确认当前版本安装包包含 `src-tauri/windows/nsis/installer-hooks.nsh`
- 确认生成的 `installer.nsi` 在复制 external binaries 前执行 `NSIS_HOOK_PREINSTALL`
- 如用户已卡在旧安装器窗口，可先取消，再重新检查更新或手动下载新版安装包

### 10.5 CI 的 Cargo check 失败

典型报错：

```text
resource path `binaries/anybridge-proxy-<target-triple>` doesn't exist
```

处理：

- CI 必须先执行 `python scripts/build/build_sidecar_plain.py --platform "$HOST"`
- `HOST` 可以通过 `rustc -vV` 读取
- 确认 `sidecar` 依赖已经 `npm ci`

### 10.6 macOS 双架构注意事项

Release workflow 当前分别构建：

- `aarch64-apple-darwin`
- `x86_64-apple-darwin`

两个架构都要有对应 sidecar，否则 Tauri 构建会失败或运行时找不到 sidecar。

## 11. GitHub Actions 成本与限制

公开仓库通常不需要担心 Actions 分钟数费用。私有仓库会消耗额度，且 macOS runner 成本最高。

减少浪费的建议：

- 普通提交只跑 CI，不要频繁打 tag。
- 本地先跑 Windows NSIS 包验证。
- Release Notes 确认无误后再打版本 tag。
- 不要移动已经发布的 tag，发现问题发新 patch 版本。

## 12. 发版最终清单

发版前：

- 版本号已同步
- `CHANGELOG.md` 有中文用户向说明
- UI 检查通过
- Rust 格式检查通过
- `cargo check` 通过
- Windows 本地 NSIS 打包通过
- NSIS hook 已进入 `installer.nsi`

发版中：

- 已提交 release commit
- 已推送 `main`
- 已推送 `v*` tag
- CI 全平台 success
- Release workflow 全 job success

发版后：

- 主仓库 Release `isDraft=false`
- Release 资产完整
- `latest.json.version` 正确
- `latest.json.notes` 无敏感或内部信息
- Windows 自动更新 URL 指向新版 `x64-setup.exe`
- 手动下载页可访问
